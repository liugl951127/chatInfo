F1

用户登录流程User Login · Login.vue → authApi → JWT → Pinia store

客户/坐席在 Login.vue 提交用户名密码,经 axios 调 /auth/login。cs-auth 用 BCrypt 校验密码,签发 HS512 JWT (24h),载荷含 uid/role/nickname。前端把 token 持久化到 Pinia + localStorage,后续 axios 拦截器自动注入 Authorization: Bearer ... 头。

泳道说明: 前端 ↔ 网关 ↔ cs-auth ↔ MariaDB ◆ = 决策点 ━━ = 错误路径
关键算法
BCrypt matches(rawPwd, dbHash) · 恒定时间比较防时序攻击
JWT Claims
uid, role, nickname, exp=24h · HS512 签名,密钥 64 字节随机
前端持久化
Pinia useUserStore → localStorage · 刷新不丢
F2

创建会话流程Create Session · Customer → pickAgent → Redis queue → DB

客户点击 FAB → 开始咨询,前端 imApi.createSession(skill) 调 POST /api/im/session/create。后端先查 Redis 在线坐席集,按 技能匹配 + 状态排除 选最优坐席,立即分配 (ACTIVE) 或进入等待队列 (WAITING) 并广播 /topic/sessions/new。整个流程 单次 DB 写 + 一次 Redis 写,平均耗时 < 30ms。

技能路由 坐席分配算法: skill_tags LIKE 匹配 → 排除 BUSY/AWAY → 取最少会话数
Redis Keys
chat:agent:online (Set) · chat:agent:session:{id} (String) · chat:queue:waiting (List)
响应时间
平均 25ms · 1 次 SQL INSERT + 2 次 Redis SET + 1 次 Pub/Sub PUBLISH
并发安全
原子 SETNX 维护 agent:session 映射,防止同坐席被分配到 2 个客户
F3

坐席抢单流程Agent Claim · CAS Atomic Update · Cross-line Prevention

多个坐席同时抢同一会话时,采用 数据库 CAS (Compare-And-Swap):UPDATE chat_session SET agent_id=X, status='ACTIVE' WHERE session_no=? AND (agent_id IS NULL OR status='WAITING')。影响行数 = 0 说明抢单失败,立即回滚 Redis 状态。关键防御:Redis 的 chat:agent:session:X 同时只能映射一个 sessionId,杜绝「同坐席串线」BUG。

⚡ 双重防护: DB CAS (rowsAffected) + Redis SETNX (原子性) · 缺一不可
CAS SQL
UPDATE ... WHERE status='WAITING' AND agent_id IS NULL · rows=0 即失败
Redis SETNX
同坐席只能 SETNX 成功一次 · 防止「双会话串线」历史 BUG
测试用例
5 坐席并发抢同一会话,只有 1 个 SETNX 成功,4 个立即看到「已被抢走」
F4

消息发送流程Message Send · STOMP SEND → MessageService → Dual Push

客户/坐席发消息:前端 stomp.publish('/app/send/'+sessionId, dto),经 STOMP 帧到 cs-im。MessageController.send 调 MessageService.handleIncoming,DB 持久化 + 三方推送:发送者 echo + 对端实时 (在线) 或离线存储 (不在线)。三层防线:在线推送 → 离线存储 → 重连补漏 (loadHistory)。

三层兜底 ① 在线推送 ② 离线 Redis List ③ 重连 loadHistory (DB 权威)
路由解耦
SimpMessagingTemplate + Redis Pub/Sub 实现跨实例推送,坐席在哪台机都收得到
幂等订阅
ws-client.js 相同 (dest, cb) 只订阅一次,解决 v2.0.0 消息双发 BUG
消息 ID
msgId 自增,前端按 id 去重,offline list 与 history merge 不会重复
F5

转人工流程Bot → Human Handoff · TransferToHumanEvent · close bot session

V3 核心场景之一:客户在与 AI Bot 对话中可一键申请「转人工」,或 AI 识别到客户情绪低落/反复无解时主动推荐转人工。后端关闭当前 bot 会话,自动选同技能空闲坐席,创建新 ACTIVE 会话,双方 STOMP 推送 TRANSFERRED 事件。

关键: Bot 会话 CLOSED 后创建新 ACTIVE 会话,保留聊天历史供坐席参考
AI 触发阈值
情绪分 < 0.3 · 连续 3 次相同问题 · 5 分钟未解决
上下文保留
Bot 会话的 chat_message 不删,新会话中坐席可滚动看到完整历史
无坐席兜底
503 错误 → 退回 Bot 会话 + 通知「当前繁忙,请稍候」
F6

视频 / 电话流程Video & Voice · init → ICE → SDP → ASR → TTS

V3 渠道层核心:视频走 WebRTC P2P(cs-video 信令服务器转发 SDP/ICE),电话走 sip.js + ASR/TTS + M3 决策。两者复用 cs-im 的会话 / 录制链路,信令通过 STOMP 复用现有 WebSocket 连接。

视频路径: P2P 直连 (NAT 穿透用 coturn) · 信令走 cs-video STOMP 电话路径: ASR → M3 决策 → TTS · 全双工 PCM 流
WebRTC ICE
STUN 失败时走 TURN 中继,确保 95%+ 穿透率
录制
视频通话复用 RecordService,WebM 分片 + 结束 ffmpeg 合并 (阶段 3)
电话 M3
Function calling: 查订单/转人工/创建工单,通话中实时决策
F7

健康分计算流程Health Score · heartbeat → ProfileService → tags → score

V3 阶段 3 核心能力:cs-customer-success 服务定时 (cron 1h) + 事件触发 (会话结束/订单完成) 双重驱动,聚合 cs-cdp 的 30+ 标签,按权重公式计算 0-100 分,落库 health_score_history 用于趋势分析。

公式: 0.3·login + 0.3·feature + 0.2·ticket + 0.2·csat · 输出 0-100
触发策略
Cron 每小时 + 会话关闭事件 + 订单完成事件 · 三源聚合
分档阈值
≥80 Champion / 60-80 Healthy / 40-60 At-Risk / <40 Churned
下游联动
At-Risk → cs-prediction 触发主动推送 · Churned → 通知 CSM
F8

预见式触发流程Predictive Trigger · event → evaluate → match → push

V3 阶段 1-2 核心能力:事件 → 规则匹配 → AI 推理 → 主动推送。前端埋点 SDK 把行为/订单事件推到 cs-cdp,cs-cdp 写入 cdp_event 后通过 Redis Stream 通知 cs-prediction,后者加载 prediction_rule 表 + 调 M3 推理,命中后通过 STOMP 推到客户端 /user/{uid}/queue/events。

5 个内置规则: ORDER_STUCK_24H · PAYMENT_FAILED_3X · SILENT_30D · BIRTHDAY_WEEK · HIGH_VALUE_RETURN
Redis Stream
cs-cdp XADD → cs-prediction XREAD · 异步解耦,失败重试
规则引擎
JSONLogic 表达式,支持 $gte / $in / 时间窗口,管理员可热更新
节流
同 user 同 rule 24h 内最多触发 1 次 · 防止打扰
